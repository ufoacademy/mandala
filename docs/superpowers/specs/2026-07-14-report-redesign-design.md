# 프리미엄 리포트 디자인/콘텐츠 개편 — 설계 문서

날짜: 2026-07-14

## 배경

현재 `report.html`은 초록/민트 브랜드 톤의 연속 스크롤형 리포트다. 사용자가 새로 제공한 샘플(`이지영_건강만다라_검진결과_통합분석_프리미엄_리포트_10페이지.pdf`)은 남색+금색 톤의 정식 "보고서" 형태로, 페이지마다 챕터 제목과 페이지 번호가 반복되고, 통계 대시보드·목표치 표 등 구조가 다르다. 이 설계는 `report.html`의 디자인과 콘텐츠 구성을 샘플에 맞춰 개편하는 것을 목표로 한다.

## 사용자 확정 사항 (브레인스토밍 중 결정됨)

1. 비주얼 톤: 샘플 그대로 남색/금색 팔레트로 완전 교체 (기존 초록 브랜드 톤 폐기).
2. 구조: 각 섹션을 독립된 1페이지로 구성하고, 페이지마다 상단에 챕터 제목("0N. ○○○")과 페이지 번호를 반복 표시하는 보고서 형식.
3. "목표치 설계" 페이지의 현재값→목표치 숫자는 기존 `BIOMARKERS[key].grade()` 임계값을 기준으로 클라이언트 JS가 계산 (AI가 숫자를 지어내지 않는다는 기존 원칙 유지).
4. 검진수치를 입력하지 않은 습관전용 사용자는, 검진데이터가 필요한 페이지(바디 시그널 대시보드, 목표치 설계)를 통째로 생략하고 나머지 페이지로만 구성. 페이지 번호는 실제 출력되는 페이지 기준으로 재부여.
5. 4개 페어 섹션(식&동/휴&심/환&연/지&락)마다 하단 하이라이트 박스의 제목과 성격을 샘플처럼 다르게 (미션/루틴제안/체크리스트/번역문장 등) — AI가 섹션 성격에 맞춰 생성.
6. 기존 "🥗 나의 영양설계 추천" 섹션(샘플엔 없음)은 삭제하지 않고 새 디자인에 맞춰 별도 페이지로 유지.

## 페이지 구성 (최대 11페이지, 검진데이터 없으면 9페이지)

| # | 페이지 | 조건 |
|---|---|---|
| 1 | 표지 — 인사말, 대상자 정보, 4개 Core Insight 카드, 리포트 소개 박스 | 항상 |
| 2 | 바디 시그널 대시보드 — 통계 카드 + 검진수치 요약표 | 검진데이터 있을 때만 |
| 3 | 건강만다라 8영역 맵 — 레이더차트 + 막대차트 + 8영역 해석 요약 | 항상 |
| 4 | 식 & 동 | 항상 |
| 5 | 휴 & 심 | 항상 |
| 6 | 환 & 연 | 항상 |
| 7 | 지 & 락 | 항상 |
| 8 | 목표치 설계 — 현재→목표 표 + 8영역 우선순위 + 희망 메시지 | 검진데이터 있을 때만 |
| 9 | 나의 영양설계 추천 | 항상 |
| 10 | 12주 실행 로드맵 + 메시징 서비스 예시 | 항상 |
| 11 | 최종 결론 | 항상 |

표지를 제외한 각 페이지는 헤더에 순번("01.", "02." …)을 표시하며, 이 순번과 우측 상단 페이지 번호는 렌더링 시점에 실제로 그려지는 페이지 목록을 기준으로 JS가 동적으로 계산한다 (하드코딩 아님).

## 스키마 변경 (`lib/reportSchema.js`)

`pairedSections` 항목 필드를 아래처럼 재정의한다 (기존 `title`/`executionTranslation`/`monthlyMission` 폐기 — 세션 저장 데이터라 하위호환 불필요):

```
{
  leftTitle, leftNarrative,      // 예: "식 · 혈당·혈압·체중", "동 · 활동 설계" + 본문
  rightTitle, rightNarrative,
  middleTitle, middleNarrative,  // 예: "실행 번역"/"상위 핵심축"/"왜 중요한가"/"강점과 보완점" + 본문
  highlightTitle, highlightContent // 예: "이번 달 핵심 미션"/"회복 루틴 제안"/"즉시 적용 체크리스트"/"프리미엄 번역 문장"
}
```

- 페이지 헤더의 "식 & 동" 등 페어 이름은 고정 순서(식&동→휴&심→환&연→지&락)이므로 AI가 생성하지 않고 `report.html`에서 인덱스로 하드코딩.
- `highlightContent`는 줄바꿈으로 구분된 3~5개 항목 문자열로 생성하도록 프롬프트에 지시하고, 렌더링 시 항상 불릿/체크리스트 형태로 표시 (페이지마다 다른 렌더러를 만들지 않고 공통 렌더러 사용).

새 최상위 필드 추가:
- `hopeMessage: string` — 8페이지 "희망 메시지" 박스용.
- `messagingExamples: string[]` (3~4개) — 10페이지 "메시징 서비스 예시" 박스용.
- `finalConclusion.finalProposal`은 줄바꿈 구분 다중 항목 문자열로 생성하도록 프롬프트 지시 (렌더링 시 번호 리스트로 표시).

기존 유지: `greeting`, `coreInsights`(4개), `areaInterpretations`(8개), `priorityExplanation`, `nutritionPlan`, `roadmap12Week`.

## 클라이언트 계산 데이터 (숫자는 AI 개입 없이 JS로만 계산)

`index.html`에 새 함수 `collectTargetTableForReport()` 추가:
- 실제로 입력된 검진 항목 중 표지에 적합한 주요 항목(체중/BMI, 허리둘레, 공복혈당, HbA1c, 중성지방, 혈압, 비타민D 등 — 존재하는 항목만)에 대해 `{ name, current, unit, target, note }`를 계산.
- `target`은 `BIOMARKERS[key].grade()`가 사용하는 임계값(정상/경계 경계선)을 근거로 계산 (예: 정상 상한 바로 아래 값 또는 정상 범위를 문자열로 표기).
- `note`는 기존 `BIOMARKERS[key].habit` 문구의 첫 구절(첫 쉼표 이전)을 재사용 — 새 데이터 추가 없이 기존 정보 재활용.
- 검진데이터가 없으면 빈 배열을 반환하고, `report.html`은 이를 근거로 2p/8p를 생략한다.

`collectCheckupSummaryForReport()`는 기존 그대로 유지 (변경 없음, 2p 대시보드 표에 재사용).

## 비주얼 시스템 (report.html 전면 재작성)

- 팔레트: 헤더 남색(`#1B2A47` 계열) + 금색 포인트 라인(`#C9A227` 계열), 본문 배경 연한 회청색(`#F5F7FA`), 인사이트 카드 연초록/연파랑 톤, 하이라이트 박스 크림색+주황 좌측 보더, 섹션 카드는 흰 배경+둥근 모서리+옅은 그림자 — 첨부 샘플 이미지를 시각적 기준으로 삼는다.
- `.report-page` 컴포넌트: `page-break-before:always`(표지 제외), 헤더(챕터 제목 + 부제 + 우상단 페이지 번호 + 하단 금색 라인), 본문, 공통 푸터(저작권/문의 문구, 기존 문구 유지).
- 기존 `drawRadar()` 캔버스 로직은 색상만 남색 팔레트로 교체해 재사용. 막대차트는 기존 `.area-bar-row` 구조를 재사용하되 색상만 교체.
- 인앱 브라우저 배너, 인쇄 버튼, `escapeHtml`, `sessionStorage` 로직은 변경 없음 — 신규 `renderReport()` 본문만 새 마크업으로 교체.

## 프롬프트 변경 (`lib/buildPrompt.js`)

시스템 프롬프트에 다음 규칙 추가:
- `pairedSections`의 각 필드(leftTitle/rightTitle/middleTitle/highlightTitle) 네이밍 가이드와 페어별 성격 예시 제공 (식&동=실행 번역, 휴&심=상위 핵심축, 환&연=왜 중요한가, 지&락=강점과 보완점 — 단, AI가 문맥에 맞게 자유롭게 표현해도 무방하다고 명시).
- `highlightContent`/`finalConclusion.finalProposal`은 줄바꿈으로 구분된 짧은 항목 3~5개로 작성.
- `hopeMessage`, `messagingExamples` 작성 가이드 추가.
- 숫자 관련 규칙(전달받은 숫자만 사용, 새로 만들지 않음)은 목표치 표에도 동일 적용 — 단 목표치 표 자체는 AI가 생성하지 않고 클라이언트가 만든 데이터를 그대로 렌더링하므로 프롬프트에는 영향 없음.

## api/generate-report.js 변경

- `pairedSections` 배열 clamp 로직(`slice(0,4)`) 유지.
- `messagingExamples` 배열이 있으면 `slice(0,4)`로 방어적 clamp 추가.
- 그 외 로직 변경 없음 (검증/웹훅/에러처리 동일).

## 에러 처리 / 제약사항

| 상황 | 처리 |
|---|---|
| AI 서술이 길어서 한 페이지 분량을 넘김 | 강제 자르지 않음 — 인쇄 시 해당 챕터가 물리적으로 2페이지에 걸쳐 출력될 수 있음을 허용. 대신 프롬프트에 각 필드별 대략적 글자수 가이드를 명시해 발생 빈도를 낮춘다 |
| 검진데이터 없는 습관전용 진단 | 2p, 8p 생략. 페이지 번호/챕터 번호는 실제 렌더링되는 페이지 기준으로 동적 계산 |
| 모바일 화면(인쇄 전) | 각 `.report-page`는 화면에서는 세로로 이어서 스크롤되는 카드 형태로 보임 (A4 강제 비율 아님), 인쇄 시에만 페이지 분할 적용 — 기존 동작과 동일 |

## 범위 밖
